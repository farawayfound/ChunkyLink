# Cross-Reference System

## Overview

The cross-reference system links related chunks across documents to improve search thoroughness. It runs as a post-processing step after indexing.

## Architecture

**Two-phase processing:**
1. **Indexing phase**: Extract and chunk documents (build_index_incremental.py)
2. **Cross-reference phase**: Build relationships across all chunks (build_cross_references.py)

## Implementation

### Core Module: utils/cross_reference.py

**Functions:**
- `expand_terms(tags)`: Expands tags with domain-specific synonyms
- `build_search_text(chunk)`: Flattens all searchable content into single field
- `compute_similarity(chunk1, chunk2)`: Uses spaCy word vectors for semantic similarity
- `find_related_chunks(target, all_chunks)`: Finds top N similar chunks from other documents
- `build_topic_clusters(all_chunks)`: Groups chunks by shared tags
- `enrich_chunk_with_cross_refs(chunk, all_chunks, clusters)`: Adds all cross-reference metadata

### Domain Synonyms (TERM_ALIASES)

**Auto-generated by default** from corpus analysis using spaCy similarity.

Optionally configure in `config.py`:
```python
TERM_ALIASES = {
    "xumo": ["xumo", "scxi11bei", "streaming device"],
    "authentication": ["auth", "login", "credential"],
}
```

If `TERM_ALIASES` is empty or not defined, the system will:
1. Analyze top 50 most frequent tags across all chunks
2. Find co-occurring terms in the same chunks
3. Use spaCy similarity (>0.6) to identify synonyms
4. Build alias groups automatically (max 5 synonyms per term)

## Added Fields

### search_keywords (List[str])
Expanded terms including all synonyms from TERM_ALIASES
```json
"search_keywords": ["xumo", "scxi11bei", "streaming device", "auth", "login", "credential"]
```

### search_text (str)
Flattened searchable content combining:
- text_raw or text
- breadcrumb
- tags
- key_phrases
- all entity values

### related_chunks (List[str])
Array of chunk IDs with semantic similarity ≥ 0.7 (configurable)
```json
"related_chunks": [
    "doc2.pdf::ch03::p15-17::para::abc123",
    "doc5.pdf::ch01::p5-8::para::def456"
]
```

### topic_cluster_id (str)
Cluster identifier based on shared tags
```json
"topic_cluster_id": "authentication+device+troubleshooting"
```

### cluster_size (int)
Number of chunks in the same cluster

## Configuration (config.py)

```python
MAX_RELATED_CHUNKS = 5          # Max related chunks per chunk
MIN_SIMILARITY_THRESHOLD = 0.7  # Minimum similarity score (0-1)
ENABLE_CROSS_REFERENCES = True  # Toggle cross-ref building

# Optional: Define domain-specific synonyms (leave empty for auto-generation)
TERM_ALIASES = {
    # "term": ["synonym1", "synonym2", ...]
}
```

## Usage

### Recommended: Combined Approach (2x faster)
```bash
cd indexers
python build_index_with_cross_refs.py
```

This processes documents and builds cross-references in one pass.

### Alternative: Two-Stage Approach
```bash
cd indexers
python scripts/build_index_incremental.py  # Index documents
python scripts/build_cross_references.py    # Build cross-references
```

Use two-stage when:
- Rebuilding cross-refs with different thresholds
- Processing >50K chunks (rare)

### Incremental Updates
```bash
cd indexers
python build_index_with_cross_refs.py  # Processes new/modified files with cross-refs
```

Or with two-stage:
```bash
cd indexers
python scripts/build_index_incremental.py
python scripts/build_cross_references_incremental.py  # Only updates new chunks
```

## Performance

**Processing time (combined approach):**
- 1,000 chunks: ~25 seconds
- 5,000 chunks: ~2 minutes
- 10,000 chunks: ~5 minutes

**Processing time (two-stage approach):**
- 1,000 chunks: ~45 seconds
- 5,000 chunks: ~4 minutes
- 10,000 chunks: ~12 minutes

**Optimizations:**
- Cached spaCy doc objects (avoid recomputation)
- Stricter tag overlap filter (≥3 shared tags)
- Chunk deduplication (content hash)
- Reduced metadata bloat
- Inline processing (no I/O between stages)

## Search Integration

### PowerShell Multi-Field Search
```powershell
# Search across expanded fields
Get-Content "$outDir\detail\chunks.jsonl" | ConvertFrom-Json | 
    Where-Object { 
        ($_.search_text -match $term) -or 
        ($_.search_keywords -contains $term)
    }
```

### Related Chunk Expansion
```powershell
# Get initial matches
$matches = # ... search results

# Expand with related chunks
$relatedIds = $matches.related_chunks | Select-Object -Unique
$related = Get-Content "$outDir\detail\chunks.jsonl" | ConvertFrom-Json |
    Where-Object { $relatedIds -contains $_.id }

# Combine
$allResults = ($matches + $related) | Sort-Object -Property id -Unique
```

### Cluster-Based Search
```powershell
# Find all chunks in same topic cluster
Get-Content "$outDir\detail\chunks.jsonl" | ConvertFrom-Json | 
    Where-Object { $_.topic_cluster_id -eq $targetClusterId }
```

## Benefits

1. **Cross-category discovery**: Finds related content even if categorized differently
2. **Synonym matching**: Searches work with term variations (auth vs authentication)
3. **Context expansion**: Related chunks provide additional context to Amazon Q
4. **Topic clustering**: Groups semantically similar content for broader searches
5. **Flattened search**: Single field contains all searchable content

## Limitations

1. **Memory usage**: Loads all chunks into memory during processing
2. **spaCy dependency**: Requires en_core_web_md model
3. **Same-document exclusion**: Related chunks exclude same document (by design)
4. **Processing time**: Scales with chunk count (optimized with caching)

## Future Enhancements

- Approximate nearest neighbors (FAISS) for faster similarity (>50K chunks)
- Co-occurrence matrix for term relationships
- Hierarchical tag taxonomy
- Persistent spaCy cache across runs
